# 微信渠道清爽输出规范（避免表格拥挤）

当用户处于微信聊天环境时，按以下规则组织回复内容（面向用户可见文本；技能文档自身的 Markdown 结构不在此约束范围内）。

## 编辑与写操作前确认（强制）

在**执行任何会改动系统或仓库状态的操作之前**，必须先向用户**说明拟执行动作**（含影响范围、关键参数：路径、环境、请求体或文件变更摘要等），并得到用户**明确同意**（例如「确认」「可以执行」「按这个来」）后，才可执行。

涵盖但不限于：创建/修改/删除仓库内文件、批量替换、会改写工作区或依赖目录的命令；以及任何会改远端/业务数据的接口调用。

**可不经确认**：只读排查、纯口头方案、仅按本技能调整**用户可见回复排版**（不写文件、不调写接口）。

若用户已在**同一条消息**中明确授权某一具体动作（含范围），可视为已确认，但执行前仍应**简短复述**将运行的命令或写入点，避免误操作。

## 核心原则

1. 回答以“解决用户当前问题”为唯一目标，不强制套固定模板。
2. 默认简洁：只给满足问题所需的最少信息，非必要不展开。
3. 若用户只问一个点，优先一句话直答；需要时再补 1-3 条关键说明。
4. 先答再展：先给结论，再按需补步骤、风险、背景。

## 微信可读性规则

1. 禁止使用 Markdown 结构符号：不要 `#`、` ``` `、`*`、`>`、`[]`、`-` 等。
2. 段落短小，每段 2-4 行，段落之间空一行。
3. 重点信息用【】标出，不用星号加粗。
4. 分点用 ✅ 📌 ⚠️ 等 emoji 区分，不用列表符号（也不要用 `•`、`-` 这类列表符号）。
5. 代码仅使用单行 `反引号`，禁止多行代码块。

## 表格化输出规则（从严，含“伪表格”）

1. 禁止 Markdown 表格（含 `|` 分隔列、表头分隔线等）。
2. 禁止“伪表格”：先输出一行表头字段名，再用多行“序号 + 多列值”对齐成表格的阅读形态。
3. 禁止单独输出 `#` 作为标题或占位（微信端常显得像 Markdown 标题残留）。
4. 禁止用大量空格把多列数据挤在同一行（看起来像终端对齐输出）。
5. 数据一多必须改为“分组短段 + 结论先行 + 分页”，不要一次性塞完整明细。

## PSI 系统场景建议

涉及 PSI 管理系统问题时，按问题复杂度动态选择：

1. 简单问答：直接给结论和一个关键动作。
2. 操作类问题：给最短步骤（通常 2-4 条）。
3. 排障类问题：给可能原因前 2-3 项和优先排查顺序。
4. 大数据结果：只展示摘要与前几条关键项，其余提示可继续展开。

## 列表数据展示规范（重点优化）

1. 严禁输出“空格平铺行”。
2. 列表必须“逐条展示”，每条独立成段，且字段名明确（如 编号、名称、状态、时间）。
3. 每条只放 3-5 个核心字段；非关键字段省略或折叠为“更多信息可展开”。
4. 超过 5 条时默认只展示前 3-5 条，并说明“还有 N 条，可按条件继续筛选”。
5. 优先输出顺序：先摘要（总数/异常数）-> 再关键项列表 -> 最后给下一步动作。

## 库存盘点/库存清单（强制策略）

当用户要“库存情况清单/盘点/库存列表”等明细类问题时，默认不要输出“完整清单”，除非用户明确说“全部/导出/完整”。

✅ 【摘要】先给总数、异常/预警数量（如有）、以及本页展示范围

📌 【本页明细】默认只展示 3-5 条，优先展示最需要关注的项（预警、负库存、低库存、高库存、最近变动等；若无异常则按仓库分组各取 1-2 条样例）

⚠️ 【分页】明确提示“还有 N 条”，并给出继续方式（例如：回复仓库名、商品关键字、只看预警、看下一页）

推荐条目格式（每条 3-5 行，字段独占一行）：

📌 【1】商品：阿迪板鞋（大童款）
仓库：晋江仓库B区
库存：1000（安全库存：10）
状态：✅ 正常
价值：如需展示再补一行（默认可省略，避免信息噪音）

推荐样式（微信友好）：

✅ 【查询结果】共 28 条，异常 3 条，已展示前 5 条

📌 【1】编号：SO20260415-001
状态：待出库
客户：华南门店A
金额：1280.00

📌 【2】编号：SO20260415-002
状态：已完成
客户：华东门店B
金额：980.00

⚠️ 还有 23 条，回复“继续”可按状态或时间范围筛选后展示
